# BloomSkill 全域開發憲章 (Main Constitution)

> [!IMPORTANT]
> ### ⚡ 高效率開發溝通規範
>
> **為了兼顧開發節奏與控制權，Agent 必須遵循以下「後置確認」邏輯：**
>
> **1. 任務驅動原則 (Issue First)：**
> - **[主動詢問]**: 不論是 Agent 發現的 Bug、技術債，或是使用者提出的新功能建議，Agent **必須**優先詢問：「是否需要為此建立 GitHub Issue 進行追蹤？」
> - **[現狀盤點]**: 在主動詢問時，必須同時利用工具列出目前專案中**所有未處理 (Open) 的 Issue 清單**，協助使用者評估優先級，無需強制引導至開發流程。
>
> **2. 檔案編輯原則：**
> - **[執行前]**: 檔案建立或修改不需要事前詢問。
> - **[執行後]**: 寫入完成後，必須完整回報「修改了哪些檔案」以及「變更摘要」。
>
> **3. Git 最終防線：**
> - **[任務完成點]**: 在一組邏輯任務全部完成後，Agent 必須條列所有預計執行的指令。
> - **[核准執行]**: 必須獲得使用者對於「指令序列」的明確授權（如回覆「執行」），方可發起 Git 相關 Tool Call。
> - **[禁止同步]**: 嚴禁在條列指令的當下直接呼叫 Git 工具。

## 第一章：PDCA 執行生命週期與自癒迴圈

本專案所有的 AI Agent 開發行為，必須遵循 **PDCA (Plan-Do-Check-Act)** 微型自動化工廠模型：

1. **[Plan / 需求定義]**: 在動作前必須輸出接收標準 (Acceptance Criteria, AC)，與使用者核對架構與邊界。
2. **[Do / 原子化開發]**: 嚴格遵守 **[RULE-006] 原子化異動限制**，單次生命週期修改檔案數建議不超過 5 個。超過請建議工程師拆分任務。
3. **[Check / 試驗與靜態掃描]**: 開發完成後必須經過靜態掃描或沙盒沙盤推演，對照 AC 是否達成。
4. **[Act / 自癒與停機處置]**:
   - 若 Check 失敗，Agent 應自動退回 Do 階段進行自我修復 (Self-Healing)。
   - **重試上限**: 單一任務自動修復上限為 **3 次**。
   - **[RULE-005] 故障停機守則**: 連續 3 次失敗或遭遇環境重大報錯，必須立即停機輸出 `[ERROR_LOG]`，等待人類介入。嚴禁嘗試繞過規範。

> [!CAUTION]
> ### ⛔ 強制停頓協議 (Mandatory Phase Gate)
>
> **每一個 PDCA 階段 (Plan / Do / Check / Act) 均為獨立任務，嚴禁 Agent 在未取得使用者明確正向授權的情況下，自主串聯執行下一個階段。**
>
> - **授權字眼包含**：「執行」、「繼續」、「進行」、「Yes」、「OK」或其他明確肯定意圖的回覆。
> - **授權必須在該階段結束後、下一階段開始前取得**，不得預先假設或推斷。
> - **違反此條款視同違背至高無上原則**，無論任何情境皆不得繞過。

## 第二章：權限邊界與逾時防護 (Safety & Timeouts)

1. **[指令隔離原則]**: 除 `git-manager` 以外的一般業務技能，**嚴禁**調用 `git`、`rm -rf` (目錄級刪除) 或 `curl` (未經授權之外部發送)。
2. **[明確與互不侵犯]**: 嚴禁在無人工介面「特定指令點擊」授權下，自作主張鏈式喚起其他高階系統管理技能。
3. **[逾時防護 (10s)]**:
   - 交互對話若超過 10 秒無反應，應自主重試。
   - 任何任務執行 (Validate/Test/Commands) 嚴格設定 **10 秒** 為最終 Timeout 限制。
   - 長時腳本必須保持 **Progress Echo**，每 3 秒印出進度以維持狀態。
4. **[文字處理原則]**: 嚴禁在 shell 指令中直接處理中文字元過濾 (grep/sed)，必須優先改用內建之 `grep_search` 工具，以避免環境編碼 (Locale) 導致的執行掛起。

## 第三章：Skill 開發規範 (The Gold Standard)

每個 Skill 資料夾必須保持完全封裝並具備：

- **SKILL.md**: 必須內含對應的「高效率開發」確認原則。
- **README.md**: 應列出該技能專屬指令與相依性。
- **scripts/**: 同時提供 `.ps1` (Win) 與 `.sh` (Mac) 之雙軌制。

## 第四章：交付與報告

- **[結構化回覆]**: 所有執行狀態的回報必須遵循 `.agent/skills/git-manager/docs/REPLY_TEMPLATE.md` 之結構，確保流程透明。
